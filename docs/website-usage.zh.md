# HireLoop 网站使用说明

最后更新：2026-05-06

本文面向实际使用网站的操作者，说明当前 `HireLoop Virtual Org Console` 的主要使用方式，以及哪些动作只是只读导航或决策支持，哪些动作仍然必须停留在后端双确认边界内。

当前网站是一个**招聘辅助决策工作台**，不是自动录用系统。

- Graph RAG 搜索、图谱分析、Reviewer 结果都只是**决策支持**。
- 顶部和候选人详情里的飞书按钮只做**安全网页跳转**。
- 任何真实写入（包括 Analytics 周报）都不能由前端直接自动执行。

## 1. 当前页面的两个主场景

页面中的 `Competition 场景切换` 提供两个主场景：

### 场景一：候选人搜索

适合做这些事：

- 用自然语言搜索候选人
- 查看候选人与岗位的匹配度
- 阅读 Graph RAG 的命中特征、邻居扩散、query-aware subgraph
- 点击候选人进入图谱深度分析

### 场景二：单简历复核

适合做这些事：

- 复核已有候选人，判断是否建议录取
- 临时粘贴新简历，结合岗位信息即时做 Reviewer RAG 判断
- 结合岗位记忆、图投影、邻居证据和 LLM Reviewer 输出做人类判断

## 2. 飞书网页联动怎么用

当前页面有两个飞书网页入口：

### 顶部：`打开飞书 Base`

- 作用：打开当前配置的飞书 Base 网页。
- 实际路径：浏览器访问 `/go/base`，由后端安全解析成真实 Feishu Web URL。
- 按钮启用条件：`/api/live/base-status` 返回 `feishuWebUrlAvailable=true`。
- 如果未配置 `FEISHU_BASE_WEB_URL` / `LARK_BASE_WEB_URL` 或表级 URL，按钮会保持禁用。

### 候选人详情：`打开飞书记录`

- 作用：从真实候选人详情抽屉跳到对应飞书记录所在的表页面。
- 实际路径：浏览器访问 `/go/:linkId`。
- 前端只持有 opaque `lnk_live_*`，不持有真实 `rec_xxx` record ID。
- 如果对应网页 URL 不可用，后端只返回安全 unavailable message。

这两个入口都只是**网页导航**，不做写入。

## 3. 场景一：候选人搜索怎么用

### 步骤 1：输入自然语言查询

在 `Graph RAG 查询` 输入框中直接输入自然语言，例如：

- `我想找一个会 Python 和 Rust 并且是本科毕业的员工`
- `找一个偏数据分析方向、熟悉 SQL、沟通能力强的人`
- `想找适合做 Product Manager 的候选人，最好有 A/B Testing 和 Roadmapping 经验`

然后按回车。

### 步骤 2：看候选人队列

查询后会显示：

- `Graph RAG 候选人队列`
- 候选人的最终分数
- 简要匹配原因
- 证据数量
- 相似候选数量

候选人队列按分数降序排列。

### 步骤 3：看检索过程

查询结果上方会显示 `检索过程`，这里可以看到：

- 数据源
- 查询模式
- 候选人池大小
- 返回数量
- 解析出来的 token
- 每个候选人的：
  - 最终分
  - 基础分
  - 检索增益
  - 命中特征节点
  - 缺失约束
  - 为什么加分 / 减分

### 步骤 4：点击候选人进入深度分析

点击任意候选人卡片，会显示：

- 图谱总览
- 岗位历史记忆
- 命中的关键特征
- 相似候选人参考网络
- 图游走过程
- 邻居扩散顺序
- query-aware subgraph 路径图

## 4. 图谱总览怎么看

当前图谱总览按层展示：

1. 岗位信息层
2. 当前简历层
3. 邻居简历层
4. 技能节点层

阅读建议：

- 先看当前简历与岗位之间的边
- 再看当前简历与邻居的边
- 最后看命中的技能和扩散关系

边越粗通常表示权重越高；激活或游走方向会用更明显的视觉提示展示。

## 5. 场景二：单简历复核怎么用

进入 `单简历复核场景` 后，有两种模式：

### 模式 A：复核已有候选人

适合：

- 系统里已经有候选人 ID
- 想直接判断是否建议录取

用法：

1. 输入 `candidateId`，例如 `CAN-002731`
2. 或点击 `带入当前候选人`
3. 点击 `开始复核`

结果区会显示：

- 建议录取 / 建议不录取 / 待人工判断
- 置信度
- 图投影
- GNN 信号
- 判断依据
- 邻居证据
- 图谱总览
- 复核游走过程

### 模式 B：填空新建简历

适合：

- 刚拿到一份新简历
- 还没有正式入库
- 想先做即时判断

核心字段：

- `Role / 目标岗位`
- `Job_Description / 岗位描述`
- `Resume / 简历正文`
- 可选：候选人显示名

点击 `新建简历并开始复核` 后，后台会做结构化抽取、embedding、邻居搜索、图投影和 Reviewer 判断。

注意：

- 这是**即时临时候选人复核**
- 不等于已经正式入库
- 结果仍应由招聘人员人工审阅

## 6. 候选人详情里有哪些动作

真实飞书候选人详情抽屉当前允许这些动作：

- `打开飞书记录`：跳到飞书网页
- `运行 Agent 预演`：deterministic dry-run，不写飞书
- `执行 Provider 预览`：需要确认短语，不写飞书
- `生成写回计划摘要`：只生成计划，不执行写入
- `加入异步图更新队列`：本地暂存，后续集中处理

当前不允许这些事：

- 在前端直接执行 `/execute-writes`
- 在前端直接执行人类决策写回
- 在前端直接执行 Analytics 报告写回

## 7. 历史查询与一键同步

在搜索场景下，搜索框中的查询会自动保存到 `历史查询与一键同步` 区块。

这里会显示：

- 最近的历史查询
- 每条查询的状态
- 查询摘要

`一键同步数据` 当前语义是：

- 统一处理本地暂存数据
- 不是自动录用
- 不是自动写入最终录用结果

## 8. 当前结果应该如何理解

无论是搜索场景还是单简历复核场景，都请把结果理解成：

- 强证据参考
- 排序建议
- 人工复核前置材料

不要把结果当成：

- 最终录用决定
- 自动拒绝结论
- 无需人工确认的执行命令

推荐阅读顺序：

1. 先看结论和分数
2. 再看命中的关键特征
3. 再看邻居证据
4. 最后看图谱总览和游走路径
5. 由招聘人员做最终判断

## 9. 常见问题

### 为什么打开页面时没有候选人队列？

这是当前设计。未输入查询前，页面保持空态，只显示搜索框和提示，不自动展示上次结果。

### 为什么查询结果里会提示“缺失约束”？

说明候选人满足了部分要求，但没有完全满足查询条件。系统会明确指出“接近，但不是完全命中”。

### 为什么有时会推荐“接近的人”，不是完全符合的人？

当前检索策略会在硬命中不足时给出最接近候选人，并把缺失项标出来，方便你在候选池较弱时仍然获得可复核对象。

### 新建简历复核失败怎么办？

常见原因：

- 岗位描述为空
- 简历正文为空
- `competition` 相关 LLM / embedding 接口失败
- 服务器网络或模型接口超时

优先检查：

- `/data/competition/.env`
- 服务器出网
- 模型接口是否可用

### Analytics 周报为什么没有在页面里直接“执行”？

这是刻意的安全边界。Analytics 写回会落到 Reports + Agent Runs，必须保留在后端双确认 + `planNonce` 路径里，而不是在浏览器里直接触发。
