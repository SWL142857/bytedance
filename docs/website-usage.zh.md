# HireLoop 网站使用说明

最后更新：2026-05-06

本文面向实际使用网站的操作者，说明如何使用当前已经部署到服务器上的 HireLoop 网站，包括：

- 场景切换
- Graph RAG 候选人搜索
- Graph RAG 深度图谱分析
- 单简历复核
- 历史查询与一键同步
- 常见问题

当前网站是一个**招聘辅助决策工作台**，不是自动录用系统。  
所有图谱、检索、Reviewer RAG 结果都只是**决策支持**，最终是否录取仍应由招聘人员人工确认。

---

## 1. 网站包含的两个核心场景

网站当前有两个主场景，可在页面中的 `Competition 场景切换` 里切换：

### 场景一：候选人搜索

适合做这些事：

- 根据自然语言要求搜索候选人
- 看候选人和需求的匹配程度
- 看 Graph RAG 的检索过程、命中特征、邻居扩散、query-aware subgraph
- 进入候选人的图谱深度分析

### 场景二：单简历复核

适合做这些事：

- 复核一个已有候选人，判断是否建议录取
- 临时粘贴一份新简历，结合岗位信息即时做 Reviewer RAG 判断
- 结合岗位记忆、候选人画像、图投影、邻居证据、LLM Reviewer 输出做人工判断

---

## 2. 场景一：候选人搜索怎么用

### 步骤 1：输入自然语言查询

在 `Graph RAG 查询` 输入框中直接输入自然语言，例如：

- `我想找一个会 Python 和 Rust 并且是本科毕业的员工`
- `找一个偏数据分析方向、熟悉 SQL、沟通能力强的人`
- `想找适合做 Product Manager 的候选人，最好有 A/B Testing 和 Roadmapping 经验`

然后按回车。

### 步骤 2：看候选人队列

查询后会显示：

- `Graph RAG 候选人队列`
- 候选人的最终分数
- 简要匹配原因
- 证据数量
- 相似候选数量

候选人队列按**分数降序排列**。

### 步骤 3：看检索过程

查询结果上方会显示 `检索过程`，这里可以看到：

- 数据源
- 查询模式
- 候选人池大小
- 返回数量
- 解析出来的 token
- 每个候选人的：
  - 最终分
  - 基础分
  - 检索增益
  - 命中特征节点
  - 缺失约束
  - 每一步为什么加分 / 减分

### 步骤 4：点击候选人进入深度分析

点击任意候选人卡片，会显示：

- 三层 / 四层图谱总览
- 岗位历史记忆
- 命中的关键特征
- 命中的特征节点列表
- 相似候选人参考网络
- 图游走过程
- 邻居扩散顺序
- query-aware subgraph 路径图

---

## 3. 三层 / 四层图谱怎么看

当前图谱总览会按层展示：

1. 岗位信息层
2. 当前简历层
3. 邻居简历层
4. 技能节点层

### 节点

不同层 / 类型的节点颜色不同：

- 岗位信息：蓝 / 绿 / 青
- 当前简历：橙
- 邻居简历：偏琥珀色
- 技能节点：紫色

把鼠标悬浮在节点上，可以看到更详细的信息。

### 边

边的含义：

- 岗位到当前简历：岗位约束 / 岗位记忆 / 图投影
- 当前简历到邻居：相似关系
- 当前简历到技能：技能命中
- 邻居到技能：共享技能
- 技能到技能：feature walk 扩散关系

边的视觉规则：

- **越粗代表权重越高**
- 流动虚线表示激活 / 游走方向
- 边不会直接穿过节点，会从节点外缘弯曲绕出

把鼠标悬浮在边上，可以看到：

- 边类型
- 权重
- 方向
- 含义说明

## 4. 场景二：单简历复核怎么用

进入 `单简历复核场景` 后，有两种模式：

### 模式 A：复核已有候选人

适合：

- 你已经在系统中有候选人 ID
- 想直接判断是否建议录取

用法：

1. 输入 `candidateId`，例如 `CAN-002731`
2. 或点击 `带入当前候选人`
3. 点击 `开始复核`

结果区会显示：

- 建议录取 / 建议不录取 / 待人工判断
- 置信度
- 图投影
- GNN 信号
- 岗位与候选人判断依据
- 个人信息与关键特征
- Top General Signals
- 邻居证据与历史判断
- 图谱总览
- 复核游走过程

### 模式 B：填空新建简历

适合：

- 你刚拿到一份新的简历
- 还没有入库
- 想先做即时判断

当前填写格式参考 `competition` 数据结构，核心字段是：

- `Role / 目标岗位`
- `Job_Description / 岗位描述`
- `Resume / 简历正文`
- 可选：候选人显示名

填完后点击：

`新建简历并开始复核`

后台会做这些事：

1. 用 LLM 抽取简历结构化画像
2. 生成技能 / 经验 / 教育等特征
3. 构建 embedding
4. 在同岗位或相近岗位中找邻居
5. 生成图投影
6. 调用 Reviewer RAG 给出建议

注意：

- 这是**即时临时候选人复核**
- 不等于已经正式入库
- 结果仍应由招聘人员人工审阅

## 5. 历史查询与一键同步

在搜索场景下，搜索框中的查询会**自动保存**到 `历史查询与一键同步` 区块。

这里会显示：

- 最近的历史查询
- 每条查询的状态
- 查询摘要

同时你可以点击：

`一键同步数据`

当前语义是：

- 对本地暂存的数据做统一处理
- 不是“自动录用”
- 不是“自动写入最终录用结果”

---

## 6. 当前结果应该如何理解

无论是搜索场景还是单简历复核场景，都请把结果理解成：

- **强证据参考**
- **排序建议**
- **人工复核前置材料**

不要把结果当成：

- 最终录用决定
- 自动拒绝结论
- 无需人工确认的执行命令

推荐使用方式：

1. 先看结论和分数
2. 再看命中的关键特征
3. 再看邻居证据
4. 最后看图谱总览和游走路径
5. 由招聘人员做最终判断

## 7. 常见问题

### 7.1 为什么打开页面时没有候选人队列？

这是现在的设计。  
未输入查询前，页面保持空态，只显示搜索框和提示，不自动显示上次结果。

### 7.2 为什么查询结果里会提示“缺失约束”？

说明候选人满足了部分要求，但没有完全满足你的查询条件。  
这是系统在明确告诉你“接近，但不是完全命中”。

### 7.3 为什么有时会推荐“接近的人”，不是完全符合的人？

因为当前检索策略会在“硬命中不足”时给出最接近候选人，并把缺失项标出来。  
这有助于你在候选池较弱时仍然获得可复核对象。

### 7.4 新建简历复核失败怎么办？

常见原因：

- 岗位描述为空
- 简历正文为空
- 后端调用 `competition` 的 LLM / embedding 接口失败
- 服务器网络或模型接口超时

如果失败，优先检查：

- `/data/competition/.env`
- 服务器出网
- 模型接口是否可用

### 7.5 为什么图谱里有很多边？

因为当前图里同时展示了：

- 岗位约束边
- 邻居相似边
- 技能命中边
- 邻居共享技能边
- 技能扩散边

建议优先看：

1. 当前简历与岗位的边
2. 当前简历与邻居的边
3. 当前简历命中的技能边

## 8. 推荐使用流程

### 如果你是在找人

1. 切到 `场景一：候选人搜索`
2. 输入自然语言查询
3. 看候选人队列
4. 点开高分候选人
5. 结合图谱证据人工筛选

### 如果你是刚收到一份简历

1. 切到 `场景二：单简历复核`
2. 选择 `填空新建简历`
3. 填 `Role / Job_Description / Resume`
4. 点击 `新建简历并开始复核`
5. 看 Reviewer RAG 给出的建议和证据

### 如果你要复核已有候选人

1. 切到 `场景二：单简历复核`
2. 输入候选人 ID 或带入当前候选人
3. 点击 `开始复核`
4. 根据证据判断是否建议录取

## 9. 当前网站的定位

当前网站是一个：

- Graph RAG 搜索工作台
- Reviewer RAG 复核工作台
- 招聘辅助决策台

不是：

- 自动录用系统
- 自动淘汰系统
- 纯静态展示页
